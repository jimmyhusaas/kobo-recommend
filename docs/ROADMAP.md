## 🔨 Phase 2 — 分析體驗修復（目標：本週）

> 使用者回饋核心：「不知道分析是不是還有效」、「工作書污染了我的興趣輪廓」

### 優先順序

#### 1. 上次分析時間 + 書單變動提示 ⭐
**問題：** 加了新書或刪了書之後，不知道現有分析是否還有效，感覺每次都要重跑  
**解法：**
- 分析頁顯示「上次分析：N 天前」
- 若上次分析後書單有變動，顯示「書單已新增 X 本 / 刪除 Y 本，建議重新分析」
- 若書單沒變動，顯示「書單無異動，分析仍有效」  

**工作量：** 小（DB 已有 `analyses.created_at`，只差前端邏輯）  
**成功標準：** 使用者不再需要「自己判斷」要不要重新分析

---

#### 2. 排除分析 Toggle ⭐
**問題：** 使用者有一類書（工作必讀）不代表個人興趣，納入分析會污染推薦結果  
**使用者確認：**
- 排除（不是每次選，而是標記一次永久生效）
- 預設所有書都納入分析
- 可以隨時取消排除

**解法：**
- DB：`books_read` 加 `exclude_from_analysis boolean DEFAULT false`
- 書單頁每本書加 toggle（小圖示，不佔空間）
- 分析 API 自動過濾 `exclude_from_analysis = true` 的書
- 推薦 API 同步過濾
- 分析頁顯示「本次分析納入 X 本（已排除 Y 本）」

**工作量：** 中（需要 DB migration + API + 前端）  
**成功標準：**
- 標記排除的書不出現在分析結果裡
- 標記後重新分析，結果明顯只反映個人興趣書單

---

#### 3. 分析圖表視覺化
**問題：** 類型分佈純文字不直觀  
**解法：** 用 CSS bar chart（不引新 library）  
**工作量：** 小  
**優先級：** 低，體驗加分但不影響核心功能

## 📅 Phase 4 — 推薦體驗強化（使用者回饋 2026-04）

> 核心痛點：推薦結果可信度低、分析與推薦感覺脫節

### 優先順序

#### 1. 推薦書籍可驗證性 ⭐⭐
**問題：** 不知道推薦的書是否真實存在，無法直接確認或購買  
**解法：**
- 每張推薦卡片加「查詢此書」按鈕
- 點擊開新分頁到博客來搜尋（最符合台灣使用者）
- 次選：Google Books 搜尋連結  
**工作量：** 小（純前端，加 URL 連結即可）

#### 2. 分析與推薦串連 ⭐⭐
**問題：** 分析頁一大段文字看不懂；不確定分析有沒有真的影響推薦結果  
**解法：**
- 推薦頁顯示「本次推薦根據 X 天前的分析」
- 分析頁改版：文字摘要 → 結構化標籤（閱讀偏好 tag、高頻主題、口味輪廓）
- 讓使用者一眼看出「Claude 認為我是什麼類型的讀者」  
**工作量：** 中（分析 prompt 輸出格式需調整）

#### 3. 評價欄位降低摩擦 ⭐
**問題：** liked/neutral/disliked 匯入語法麻煩，幾乎沒人用  
**解法：** 書單每本書加快速評分 emoji（👍 / 😐 / 👎），一鍵點選  
**工作量：** 小

## 開發原則

1. **先修破窗** — 影響核心使用流程的問題優先
2. **輕量優先** — 能用 CSS 解決的不引新 library
3. **單次 commit 單一功能** — 方便 rollback
4. **每個功能上線前先在本機完整測試**
