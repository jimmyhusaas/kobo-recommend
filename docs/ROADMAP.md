## Phase 1 — 核心體驗修復（目標：2 週內完成）

> 原則：先把「用了會生氣」的問題修掉，再加新功能

### P0 — 本週完成

#### 1. 推薦歷史記錄頁
**問題：** 推薦結果關掉瀏覽器就消失，使用者無法回顧  
**解法：** 新增 `/history` 頁面，列出所有歷史 batch，可展開查看每批推薦內容  
**工作量：** 小（DB 資料已存，只差前端 + GET API）  
**成功標準：** 使用者隔天回來還能看到上次的推薦

#### 2. 書籍刪除功能
**問題：** 匯入錯了無法修正，書單越來越髒  
**解法：** 書單列表每一行加刪除按鈕，呼叫 `DELETE /api/books/:id`  
**工作量：** 小  
**成功標準：** 可以刪除單筆書籍，刪後立即更新列表

#### 3. 重複匯入防護
**問題：** 同一本書可以重複匯入，分析結果被污染  
**解法：** 匯入時以 `(user_id, title)` 做 unique check，重複的跳過並在回傳中告知  
**工作量：** 小  
**成功標準：** 重複匯入時提示「已存在 X 本，跳過」

---

### P1 — 下週完成

#### 4. 自訂推薦方向
**問題：** 硬寫的 6 個方向不夠用，限制了推薦的靈活性  
**解法：** 保留現有選項作為「快速選擇」，下方加一個自由輸入框，可額外補充方向  
**工作量：** 小  
**成功標準：** 使用者可以輸入「台灣本土作品」或「科幻小說入門」並得到對應推薦

#### 5. 推薦書一鍵加入已讀
**問題：** 讀完推薦的書後要回首頁手動貼，流程斷裂  
**解法：** 推薦卡片加「已讀完 ✓」按鈕，點擊後直接呼叫 `POST /api/books` 將該書加入書單  
**工作量：** 小  
**成功標準：** 從推薦頁直接標記已讀，書單頁即時反映

---

## Phase 2 — 體驗提升（目標：Phase 1 完成後 2 週）

#### 6. 分析圖表視覺化
**問題：** 類型分佈純文字不直觀  
**解法：** 用 CSS 做簡單 bar chart（不引入 chart library，保持輕量）  
**工作量：** 中

#### 7. 上次分析時間 + 書單變動提示
**問題：** 使用者不知道分析是否還有效  
**解法：** 分析頁顯示「上次分析：3 天前，距上次分析後新增 5 本書」，提示需要重新分析  
**工作量：** 小

#### 8. 書單搜尋 / 篩選
**問題：** 書單超過 30 本後很難找到特定書  
**解法：** 頁面頂部加搜尋框，前端 filter（不需要 API）  
**工作量：** 小

---

## Phase 3 — 進階功能（暫不排期，視使用狀況決定）

- Book metadata 查詢（封面圖、出版年份）via Open Library
- 作者國籍自動驗證（取代目前靠 LLM 猜）
- API 成本追蹤（每次呼叫花多少錢的明細）
- 書單匯出（CSV / 純文字）
- 多裝置同步（目前已 multi-user ready，加 auth 即可）

---

## 開發原則

1. **先修破窗** — P0 的問題不修，P2 的功能加了也沒人用
2. **輕量優先** — 能用 CSS 解決的不引新 library
3. **單次 commit 單一功能** — 方便 rollback
4. **每個功能上線前先在本機完整測試**
